## 1. Введение

В современном ритме жизни человеку сложно формировать и удерживать полезные привычки: чтение, спорт, здоровое питание, изучение языков. Психологические исследования показывают, что для формирования устойчивой привычки требуется регулярное повторение в течение нескольких недель. Без внешнего инструмента контроля большинство людей бросает начатое уже через пару дней.

Цель данной работы — разработать веб-приложение, которое помогает пользователю отслеживать ежедневное выполнение привычек, поддерживает его мотивацию через систему серий (стриков) и челленджей, а также предоставляет визуальную статистику прогресса.

Приложение реализовано как полностью функциональное веб-решение со всеми атрибутами современного продукта: авторизация пользователей, адаптивный интерфейс, тёмная тема, экспорт данных.

## 3. Анализ предметной области и аналогов

### 3.1. Предметная область

Привычка — это автоматизированное поведение, формируемое путём многократного повторения. В психологии выделяют несколько этапов формирования привычки:

1. Сигнал — что-то напоминает о необходимости выполнить действие.
2. Действие — собственно выполнение.
3. Награда — позитивное подкрепление.

Цифровые трекеры привычек выполняют сразу две роли: сигнала (через визуальный список) и инструмента подкрепления (через стрики, бейджи, прогресс-бары).

### 3.2. Анализ аналогов

| Приложение | Сильные стороны | Недостатки |
|------------|-----------------|------------|
| Habitica | Игровая механика (RPG), сообщество | Перегруженный интерфейс, сложен для новичка |
| Loop Habit Tracker | Чистый интерфейс, открытый код | Только Android, нет облачной синхронизации |
| Streaks (iOS) | Простота, красивый UI | Только iOS, платная |
| HabitNow | Гибкие напоминания | Реклама в бесплатной версии, перегружен функциями |

### 3.3. Выводы и решения для проекта

На основе анализа были приняты следующие проектные решения:

- Минималистичный интерфейс, главный акцент на «отметить за один тап».
- Стрики — ключевой мотивационный механизм (по аналогии с Duolingo и Streaks).
- Заранее заготовленные шаблоны привычек, чтобы новый пользователь начал работу за секунды, без долгого заполнения форм.
- Веб-приложение работает с любого устройства — устраняется недостаток платформозависимых аналогов.

## 4. Обоснование выбора технологий

### 4.1. Backend: PHP + Laravel

Выбор обусловлен следующими соображениями:

- **Зрелость экосистемы.** Laravel — один из самых популярных PHP-фреймворков, обширная документация, готовые решения для авторизации, миграций, валидации, очередей.
- **Скорость разработки.** ORM Eloquent, встроенный билдер запросов, шаблонизатор Blade ускоряют разработку.
- **Низкий порог развёртывания.** PHP-приложение запускается на любом shared-хостинге.

Альтернативы: Node.js + Express (требует больше ручной работы), Django (Python, был бы более громоздким для данной задачи).

### 4.2. Reactive UI: Livewire 3

Livewire позволяет писать реактивные UI без JavaScript-фреймворка. Логика остаётся на сервере (PHP), а Livewire через AJAX передаёт состояние и патчит DOM.

Плюсы для данного проекта:
- Не нужно поднимать отдельный API.
- Не нужно писать клиент на React/Vue — экономия времени.
- Один язык на всём стеке (PHP).
- Безопасность по умолчанию: вся валидация на сервере.

Альтернативы: SPA на React/Vue + REST API. В рамках курсовой это удвоило бы объём кода без выигрыша в качестве.

### 4.3. UI Framework: Flux UI

Flux — официальная библиотека компонентов от создателей Livewire. Использование Flux дало:
- Готовые доступные компоненты (Button, Input, Modal, Dropdown, Table).
- Единый визуальный язык.
- Поддержку тёмной темы из коробки.

### 4.4. Стилизация: Tailwind CSS 4

Tailwind — utility-first CSS-фреймворк. Tailwind v4 не требует config-файла, всё описывается в CSS через `@theme`, `@source`, `@custom-variant`.

### 4.5. База данных: SQLite

Для курсовой работы выбран SQLite — встроенная файловая БД:
- Не требует отдельного сервера.
- Работает на любой машине.
- Полностью поддерживается Laravel.

При необходимости конфигурация позволяет переключиться на MySQL/MariaDB/PostgreSQL без изменения кода — все запросы абстрагированы через Eloquent.

### 4.6. Сборщик фронтенда: Vite

Vite — современный сборщик с быстрой горячей перезагрузкой. Используется через `laravel-vite-plugin`, который автоматически связывает Laravel-маршруты ассетов с Vite.

### 4.7. Графики: Chart.js

Lightweight-библиотека для построения графиков на canvas. Используется для отображения процента выполнения по дням и распределения отметок по привычкам.

## 5. Архитектура приложения

### 5.1. Общий подход

Приложение реализовано как монолитное веб-приложение со следующими слоями:

1. **Слой представления (View)** — Blade-шаблоны и Livewire-компоненты, формирующие HTML.
2. **Слой контроллеров (Livewire-компоненты)** — обработчики действий пользователя, связь между UI и бизнес-логикой.
3. **Слой бизнес-логики (Services)** — чистые классы со специфическими расчётами (стрики, статистика, прогресс челленджей).
4. **Слой моделей (Models)** — Eloquent-модели, инкапсулирующие работу с БД.
5. **Слой доступа (Policies)** — проверка прав доступа к ресурсам.
6. **Слой данных (Database)** — SQLite-база, миграции.

### 5.2. Поток данных при действии пользователя

Рассмотрим типичный сценарий: пользователь нажимает «Отметить» на привычке.

1. Пользователь кликает кнопку.
2. Livewire отправляет AJAX-запрос с состоянием компонента `Dashboard` и именем метода `toggleToday(habitId)`.
3. Сервер вызывает метод компонента.
4. Компонент проверяет права через `Policy::update`.
5. Если права есть — создаёт или удаляет запись в `habit_logs`.
6. Livewire рендерит компонент заново, считает разницу с предыдущим состоянием.
7. Браузер получает HTML-патч и применяет его через morphing DOM (без перезагрузки страницы).

### 5.3. Структура каталогов

```
app/
├── Http/Controllers/             — REST-контроллеры (только StatisticsExportController)
├── Livewire/
│   ├── Dashboard.php             — главная страница
│   ├── Habits/                   — CRUD привычек
│   ├── Challenges/               — CRUD челленджей
│   ├── Statistics/Charts.php     — статистика
│   ├── Forms/                    — Livewire form-объекты
│   ├── Layout/UserMenu.php       — меню профиля в шапке
│   └── Actions/Logout.php        — действие выхода
├── Models/                       — Eloquent-модели
├── Policies/                     — проверка прав
├── Services/                     — бизнес-логика
└── Providers/                    — регистрация сервисов

resources/
├── css/app.css                   — Tailwind + Flux + кастомные стили
├── js/app.js                     — точка входа JS
└── views/                        — Blade-шаблоны

database/migrations/              — версионирование БД
routes/web.php                    — маршруты
```

### 5.4. Принципы организации кода

- **Single responsibility.** Сложная логика (расчёт стрика, статистика) вынесена в отдельные сервисы, а не пишется в Livewire-компонентах.
- **Dependency injection.** Сервисы внедряются через контейнер (typed-параметры в `render()`), регистрируются как `scoped` — один экземпляр на запрос.
- **Eager loading.** Везде где загружаются связанные сущности — явно указывается `with()`, чтобы избежать N+1.
- **Authorization through policies.** Каждое действие, связанное с владением ресурсом, проходит проверку Policy.

## 6. Структура базы данных

### 6.1. Таблицы

#### Таблица `users`

Стандартная таблица Laravel Breeze. Хранит зарегистрированных пользователей.

| Поле | Тип | Назначение |
|------|-----|------------|
| id | bigint, PK | Идентификатор |
| name | string | Имя пользователя |
| email | string, unique | Email для входа |
| email_verified_at | timestamp, nullable | Время подтверждения email |
| password | string | Хеш пароля (bcrypt) |
| remember_token | string, nullable | Токен «запомнить меня» |
| timestamps | timestamp | created_at, updated_at |

#### Таблица `habits`

Привычки пользователя.

| Поле | Тип | Назначение |
|------|-----|------------|
| id | bigint, PK | Идентификатор |
| user_id | bigint, FK → users.id, cascade | Владелец |
| title | string | Название |
| description | text, nullable | Описание |
| color | string(7), nullable | Цвет в HEX (#RRGGBB) |
| time_of_day | string(16), default 'any' | Время дня: any/morning/day/evening |
| is_active | boolean, default true | Активна ли привычка |
| timestamps | timestamp | |

Индексы: `[user_id, is_active]` — для быстрой выборки активных привычек пользователя.

#### Таблица `habit_logs`

Записи о выполнении привычек.

| Поле | Тип | Назначение |
|------|-----|------------|
| id | bigint, PK | Идентификатор |
| habit_id | bigint, FK → habits.id, cascade | Привычка |
| completed_on | date | Дата выполнения |
| timestamps | timestamp | |

Индексы:
- Уникальный `[habit_id, completed_on]` — нельзя дважды отметить привычку в один день. Это ключевое архитектурное решение, гарантирующее корректность статистики на уровне БД.
- Обычный `completed_on` — для быстрых выборок по диапазону дат.

#### Таблица `challenges`

Челленджи пользователя.

| Поле | Тип | Назначение |
|------|-----|------------|
| id | bigint, PK | Идентификатор |
| user_id | bigint, FK → users.id, cascade | Владелец |
| title | string | Название |
| description | text, nullable | Описание |
| type | enum('streak', 'total_completions', 'period_days') | Тип цели |
| target_value | unsigned int | Целевое значение |
| starts_on | date | Дата начала |
| ends_on | date, nullable | Дата окончания |
| is_active | boolean, default true | Активен ли |
| timestamps | timestamp | |

Индексы: `[user_id, is_active]`.

#### Таблица `challenge_habit` (pivot)

Связь many-to-many между челленджами и привычками. Один челлендж может включать несколько привычек, одна привычка может участвовать в нескольких челленджах.

| Поле | Тип | Назначение |
|------|-----|------------|
| id | bigint, PK | |
| challenge_id | bigint, FK → challenges.id, cascade | |
| habit_id | bigint, FK → habits.id, cascade | |

Уникальный индекс `[challenge_id, habit_id]` — одна привычка не дублируется в челлендже.

### 6.2. Связи между таблицами

- `User → Habit` — один-ко-многим (HasMany). У пользователя много привычек.
- `Habit → HabitLog` — один-ко-многим. У привычки много отметок.
- `User → Challenge` — один-ко-многим. У пользователя много челленджей.
- `Challenge ↔ Habit` — многие-ко-многим через `challenge_habit`.

### 6.3. Каскадное удаление

При удалении пользователя автоматически удаляются его привычки, отметки и челленджи. Это реализовано через `cascadeOnDelete()` во всех foreign keys. Гарантирует целостность БД и упрощает функцию «удалить аккаунт».

### 6.4. Обоснование схемы

- **Отметка не имеет поля «выполнено/не выполнено».** Сама запись в `habit_logs` означает «выполнено». Отсутствие записи — «не выполнено». Это упрощает запросы и экономит место.
- **Время дня — строка, а не отдельная таблица.** Значений всего 4, нормализация избыточна. На прикладном уровне используются константы класса `Habit`.
- **Pivot для челленджей и привычек.** Если у челленджа нет привязанных привычек — он считается распространяющимся на все активные привычки пользователя. Это упрощает создание челленджа «общего типа».

## 7. Описание модулей и компонентов

### 7.1. Модели (app/Models)

**`User`** — стандартная модель Laravel с расширением: связи `habits()` и `challenges()`.

**`Habit`** — содержит:
- Связи: `user()` (BelongsTo), `logs()` (HasMany), `challenges()` (BelongsToMany).
- Константы для времени дня и цвета по умолчанию.
- Scope `active()` для фильтра по `is_active`.
- Accessor `color` — возвращает дефолтный цвет, если в БД NULL.
- Метод `isCompletedOn(string $date): bool` — проверка отметки на дату.
- Статический метод `timeWeight()` — для сортировки по времени дня.

**`HabitLog`** — связь `habit()`, cast `completed_on` в `date`.

**`Challenge`** — связи `user()` и `habits()`. Константы типов челленджей. Метод `appliesToAllHabits()` с оптимизацией для уже загруженных через eager loading связей.

### 7.2. Сервисы (app/Services)

**`HabitStatsService`** — расчёт статистики:
- `dailyCompletion($user, $days)` — процент выполнения по дням за период.
- `perHabitBreakdown($user, $days)` — количество отметок по каждой привычке.
- `summary($user)` — сводка для дашборда.
- `heatmap($user, $weeks)` — данные для тепловой карты активности.

Все методы оптимизированы: один SQL-запрос вместо N+1.

**`ChallengeProgressService`** — расчёт прогресса челленджа в зависимости от типа:
- `calculateStreak()` — максимальная серия дней с отметками.
- `calculateTotalCompletions()` — общее число отметок.
- `calculatePeriodDays()` — число уникальных дней с отметками.

Использует eager-loaded данные если возможно, чтобы избежать дополнительных запросов.

**`UserStreakService`** — расчёт личного стрика:
- `current($user)` — текущая серия дней подряд. Если сегодня нет отметок, но вчера были — стрик ещё «жив».
- `longest($user)` — максимальная серия за всё время.
- Кэш в памяти на время запроса: оба метода переиспользуют один SQL-запрос дат.
- Ограничение глубины: загружаются только последние 2 года данных, чего достаточно для любого реалистичного стрика.

**`HabitPresetService` и `ChallengePresetService`** — каталоги готовых шаблонов:
- 24 шаблона привычек по 6 категориям (Здоровье, Спорт, Продуктивность, Учёба, Питание, Осознанность).
- 6 шаблонов челленджей разных типов и длительности.

**`MotivationService`** — выбирает мотивационную фразу в зависимости от состояния пользователя (большой стрик / 100% сегодня / новый пользователь и т.п.).

### 7.3. Livewire-компоненты (app/Livewire)

**`Dashboard`** — главная страница. Показывает огонёк стрика, мотивационный текст, прогресс дня, список привычек на сегодня (отсортированный по времени дня), сводную статистику. Метод `toggleToday($habitId)` отмечает или снимает отметку.

**`Habits\Index`** — список привычек с пагинацией. Метод `delete($id)` — с подтверждением.

**`Habits\Form`** — создание и редактирование привычки. При создании показывает каталог шаблонов с переключением категорий. Есть «Дополнительные параметры» (свернуты по умолчанию).

**`Challenges\Index`** — список челленджей с прогресс-барами. Завершённые подсвечиваются зелёным.

**`Challenges\Form`** — создание и редактирование челленджа. Поддерживает выбор шаблона, типа цели, периода и связанных привычек.

**`Statistics\Charts`** — графики и тепловая карта. Период переключается через select, графики перерисовываются через Livewire-эвент. Кнопка экспорта CSV.

**`Layout\UserMenu`** — выпадающее меню в шапке: профиль, тема, выход.

### 7.4. Политики (app/Policies)

**`HabitPolicy`** и **`ChallengePolicy`** — стандартные методы `view`, `update`, `delete`. Все возвращают `$user->id === $resource->user_id`. То есть пользователь может работать только со своими записями.

Применяются через `$this->authorize($action, $model)` в каждом методе Livewire-компонента, который изменяет данные.

### 7.5. Контроллеры (app/Http/Controllers)

В приложении почти нет контроллеров — их роль выполняют Livewire-компоненты. Единственный классический контроллер:

**`StatisticsExportController`** — стримит CSV-файл с историей отметок. Использует `lazy(500)` для защиты от переполнения памяти при большом количестве записей. Добавляет UTF-8 BOM для корректного отображения кириллицы в Excel.

## 9. Безопасность

### 9.1. Авторизация и пароли

- Авторизация через сессии Laravel (стандарт для веб-приложений).
- Пароли хешируются алгоритмом bcrypt (12 раундов по умолчанию). Соль — случайная, встроена в хеш.
- Восстановление пароля через email с одноразовым подписанным токеном.
- Подтверждение email через подписанные ссылки.

### 9.2. Защита от CSRF

Все формы Livewire автоматически защищены CSRF-токеном. Токен встраивается в заголовок страницы (`<meta name="csrf-token">`) и проверяется на каждом запросе.

### 9.3. Авторизация ресурсов

Каждое действие, связанное с владением ресурсом, проходит проверку через Policy:

```php
$this->authorize('update', $habit);
```

Если пользователь попытается подменить ID в URL или Livewire-запросе, чтобы получить чужую привычку — Policy вернёт `false`, и Laravel выбросит 403 Forbidden.

### 9.4. Защита от SQL-инъекций

Используется Eloquent ORM. Все запросы выполняются через подготовленные выражения (PDO). Прямой ввод пользователя никогда не вставляется в SQL-строку.

### 9.5. Защита от XSS

Blade автоматически экранирует все выводимые переменные через `{{ $variable }}`. Сырой HTML вывести можно только явно через `{!! $variable !!}`, что в проекте нигде не используется.

### 9.6. Валидация на сервере

Все входные данные проходят валидацию на стороне сервера через атрибуты Livewire:

```php
#[Validate('required|string|min:2|max:255')]
public string $title = '';
```

Клиентская валидация (через атрибуты HTML5) — только UX-помощник, она не определяет, что попадёт в БД.

### 9.7. Уникальность отметки

На уровне БД установлен уникальный индекс `[habit_id, completed_on]`. Это защищает от ситуации двойной отметки даже при race condition (например, если пользователь быстро кликнул дважды).

## 10. Производительность и оптимизации

### 10.1. Оптимизация запросов к БД

**Eager loading.** На странице челленджей привычки подгружаются одним SQL-запросом для всех челленджей сразу:

```php
->with(['habits' => fn ($q) => $q->where('is_active', true), 'user:id'])
```

Без этого был бы N+1: один запрос на челленджи + N запросов на привычки.

**Один запрос вместо N.** В `HabitStatsService` метод `dailyCompletion()` выполняет один запрос на все логи периода и группирует данные в PHP. Раньше было N×D запросов (N привычек × D дней).

**Индексы.** Добавлены составные индексы `[user_id, is_active]` на `habits` и `challenges`, индекс `completed_on` на `habit_logs`. Это ускоряет выборки в 10–100 раз на больших объёмах данных.

### 10.2. Оптимизация бизнес-логики

**Кэш в памяти.** `UserStreakService` хранит загруженные даты в private property. `current()` и `longest()` переиспользуют один и тот же запрос.

**Ограничение объёма выборки.** `UserStreakService` загружает только последние 2 года данных. Реалистичный стрик не может быть длиннее года — экономим память.

**`scoped` сервисы.** Зарегистрированы как scoped в DI-контейнере — один экземпляр на запрос. Кэш внутри сервисов работает корректно.

### 10.3. Оптимизация фронтенда

**Lazy-load Chart.js.** Библиотека Chart.js (~80 KB) загружается только на странице статистики через динамический `import()`. Все остальные страницы её не получают.

**Один listener на весь документ.** Spotlight-эффект использует event delegation: один `mousemove` на `document` находит ближайшую `.spotlight-card`. Не создаётся отдельный listener для каждой карточки, не пересоздаётся при Livewire-обновлениях.

**`passive: true`.** mousemove-listener помечен как passive, не блокирует прокрутку.

**CSS-only анимации.** Aurora, Star Border — на чистом CSS, без JavaScript. Аппаратное ускорение через GPU (transform, opacity).

**`prefers-reduced-motion`.** Все анимации отключаются для пользователей с этой настройкой.

### 10.4. Оптимизация экспорта CSV

`StatisticsExportController` использует `lazy(500)` — итератор, который выбирает данные пачками по 500 записей и сразу записывает в HTTP-поток. Память не растёт независимо от количества отметок — ни 100, ни миллион записей не приведут к out-of-memory.

### 10.5. Защита от двойного сабмита

Кнопки сохранения в формах привычек и челленджей помечены `wire:loading.attr="disabled"`. Пока запрос обрабатывается, кнопка заблокирована и показывает «Сохранение...».

## 11. Тестирование

### 11.1. Подход

В рамках курсовой работы проведено ручное тестирование основных пользовательских сценариев. Автоматизированные тесты предусмотрены архитектурой: PHPUnit конфигурирован, есть тестовая база (in-memory SQLite).

### 11.2. Сценарии ручного тестирования

| № | Сценарий | Ожидаемый результат |
|---|----------|---------------------|

| 6 | Создание челленджа из шаблона | Тип, целевое значение и даты заполняются |
| 7 | Расчёт прогресса челленджа | Прогресс соответствует фактическим данным |
| 8 | Стрик после нескольких дней | Текущая серия = количеству дней с отметками |

| 13 | Переключение тёмной темы | Все страницы корректно отображаются в тёмной теме |
